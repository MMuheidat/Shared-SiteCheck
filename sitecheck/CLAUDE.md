## How this project is being developed (working style)

The user is an evaluator building automated evidence for a UAE-government website audit. The workflow per pillar is: the user runs a pillar against real entity sites (TAMM, ADMO, DEWA‚Ä¶), reviews the per-question screenshots **and the pillar screen-recording video**, then gives question-by-question feedback ("Q5 should highlight the whole panel", "Q9 should actually click and verify"). Claude then plans (plan mode ‚Üí user confirms), implements, and **verifies end-to-end against the real sites before reporting back** ‚Äî visual inspection of the produced screenshots/webm is expected, not just green typechecks. Evidence quality is the product: journeys must look human (1s highlight-holds before clicks, slow wheel-scroll verification, captions narrating each question, explicit narration when a control does not exist).

### Verification workflow (no test suite ‚Äî drive the real engine)

The run APIs are auth-gated and ownership-scoped, and engine TS can't be run directly with Node (`@/` aliases, no tsx/ts-node). Verified pattern:
1. Create a temp user with `bcryptjs` via `node -e` + `@prisma/client` (upsert on a `@local.test` email).
2. `npm run dev` in background (it daemonizes ‚Äî see Commands).
3. Login via NextAuth REST with curl: `GET /api/auth/csrf` (cookie jar) ‚Üí `POST /api/auth/callback/credentials` with `csrfToken`/`email`/`password`/`json=true`.
4. `POST /api/audit/create` (`websiteUrl`, `entityName` required) ‚Üí `POST /api/audit/run-pillar` (`auditJobId`, `pillarName` must exactly match `PILLAR_CHECKS[].name`).
5. Poll `prisma.criterionResult.count` for the job until it reaches the pillar's question count (pillar 2 = 7; ~2 min/run). Beware: `console.log(number)` emits ANSI colors ‚Äî compare with `String()`/`process.stdout.write`.
6. Inspect notes + screenshots + `pillarN.webm` visually (Read tool renders PNGs).
7. Cleanup: delete the temp audit jobs (cascade), temp user, `public/screenshots/<jobId>/`, and `taskkill` the dev server PID. The user's own audits must stay untouched.

## Architecture

SiteCheck is a Next.js 16 (App Router) tool that automates a website compliance/UX audit (11 pillars, ~95 criteria in the registry; "118 points" is nominal branding ‚Äî the real denominator is computed dynamically and the naive sum of scored maxima is 100) against a target URL, primarily for UAE government-adjacent entities (see criteria referencing Tamm).

**Flow:** `app/audit/new` form ‚Üí `POST /api/audit/create` persists an `AuditJob` (status `pending`; **does not auto-run**) ‚Üí redirect to `/audit/[id]/results`, which is the action hub: "Run All Pillars" ‚Üí `POST /api/audit/run` ‚Üí fire-and-forget `lib/engine/index.ts#runEvaluation` (**non-beta pillars only** ‚Äî see below); per-pillar Run/Re-run buttons ‚Üí `POST /api/audit/run-pillar` ‚Üí `runSinglePillar`. The client then **polls** `GET /api/audit/[id]` every 5s until `complete`/`partial`/`failed`. A separate SSE route (`app/api/audit/[id]/stream`, DB-polling every 2s) feeds `/audit/[id]/progress`, but that page is only linked from the dashboard while a job is `running` ‚Äî the primary flow never visits it. PDF via `GET /api/audit/[id]/pdf` ‚Üí `lib/pdf-generator.ts` (Puppeteer renders an HTML string; screenshots inlined as base64; Arabic via `dir="rtl"` + system fonts).

**Scoring registry (`lib/scoring.ts`):** `CRITERIA` is the single source of truth for every question (`qid`, EN/AR names, pillar, `maxScore`, `isScored`, `isAutomatic`/`checkType` auto|manual|openended, `dependsOn`/`dependsOnValue` for conditional branches). `calculateTotalScore`/`calculatePillarScores` exclude `na`/`skipped`/`pending` results and sum only `isScored` criteria; grade bands: ‚â•90 Excellent, ‚â•75 Good, ‚â•60 Satisfactory, else Needs Improvement. The scoring functions do NOT evaluate `dependsOn` ‚Äî pillar engines decide skips and emit `skipped`/`na`. `lib/scoring-old.ts` is superseded ‚Äî don't extend it.

**Pillar engine (`lib/engine/`):** `PILLAR_CHECKS` in `index.ts` is the ordered registry `{name, nameAR, fn, beta?, record?, skipInitialNav?}` ‚Äî the `name` string is the join key against `CRITERIA[].pillar` and DB rows. Pillars flagged `beta: true` (currently 9 Live Chat and 10 Enquiry ‚Äî LLM integration pending) are **excluded from "Run All Pillars" and from the completeness check**, but stay runnable individually via run-pillar. `record: '<slug>'` opts a pillar into screen-recording evidence (**currently pillars 1 and 2**); `skipInitialNav: true` skips the engine's pre-pillar visit to the target URL (pillar 1 sets it ‚Äî its journey starts on a search engine; pillar 2 does NOT set it so the site opening lands on video). One module per pillar (`pillar1-discovery.ts` ‚Ä¶ `pillar10-enquiry.ts`), each exporting a `PillarCheckFn` (`lib/types.ts`) taking a params object `{page, url, auditJobId, entityName, acronym, previousResults, recorder?}` (`previousResults` feeds cross-pillar conditionals, e.g. Q36 depends on Q23; `acronym` is the optional evaluator-supplied acronym from the new-audit form) and returning `CriterionResult[]`. Browser launch tries system Chrome ‚Üí Edge ‚Üí bundled Chromium, always headless (Windows Smart App Control blocks the bundled headless shell). Pillars run sequentially; the engine re-navigates to the target URL before each pillar; a pillar that throws gets a fabricated score-0 `fail` row for every criterion in that pillar. Results are persisted per-criterion (`prisma.criterionResult.create`) as they complete. `runSinglePillar` first `deleteMany`s that pillar's rows, loads other pillars' rows as `previousResults`, and sets job status `complete` only when every **non-beta** pillar is present and nothing is `pending` (else `partial`). New pillars/questions must be registered in `CRITERIA` (scoring.ts), `PILLAR_CHECKS` (engine/index.ts), the hardcoded list in `app/api/audit/pillars/route.ts` (a deliberate duplicate to avoid importing engine deps into the route; carries the same `beta` flags for the UI), **and** the `PILLAR_NUMBERS` map in `components/ResultsTable.tsx` (pillar name ‚Üí ordinal for the "Pillar N Video ‚Äî <name>" button label).

**Evidence & video-journey conventions:** shared helpers in `lib/engine/helpers.ts` (`navigateAndWait`, `takeScreenshot`, `takeElementScreenshot`, `takeHighlightedScreenshot` ‚Äî red outline + label, `dismissCookieBanner`, and `openNavMenu`/`BURGER_MENU_SELECTORS` ‚Äî opens hamburger/3-dash navigation drawers, tag-agnostic since sites like ADMO use `<label class="menuburger">`; verifies the drawer opened by counting newly clickable links). Screenshots go to `public/screenshots/<auditJobId>/q<qid>.png` (served statically); `CriterionResult.screenshotPath` stores the **relative** `/screenshots/...` path and is a **single string** ‚Äî multi-shot checks (Q6, Q14) write extra files (`q6_page1.png`) that exist on disk but aren't referenced in the DB. **Per-pillar screen recording (`lib/engine/recording.ts`):** a pillar with a `record` slug runs in a dedicated context with `recordVideo`; the `EvidenceRecorder` handed to the pillar fn adds a click "ripple", `clickWithHighlight` (red outline held before clicking ‚Äî **pass `holdMs: 1000` for the standard 1-second deliberate hold**), a persistent `setCaption` banner (re-applied after each navigation; used to narrate every question, e.g. "Q6 ‚Äî Opening the navigation menu‚Ä¶", and to state on video when a control does NOT exist), `stamp()` for `[video @ m:ss]` notes appended to result notes, and the shared `humanScrollVerify(page)` ‚Äî slow ~300px wheel steps with ~400ms delays down the page then back to top, used to "verify like a human" after every state change (language switch, text resize, theme change). All recorder usage is optional-chained and slow gestures are gated on `recorder` presence so non-recorded runs stay fast; evidence cosmetics must never fail a criterion. On finish the video lands at `/screenshots/<auditJobId>/<slug>.webm` and the engine stamps that path onto **every** result row of the pillar via `CriterionResult.videoPath`. Pillar 9's `conductChatSession` predates this system and still opens its own `recordVideo` context, saving `livechat.webm` into `screenshotPath`. The results UI (`components/ResultsTable.tsx` ‚Üí `ScreenshotModal.tsx`) shows a per-pillar "Pillar N Video ‚Äî <name>" bar under each recorded pillar's rows; the modal renders `.webm`/`.mp4` paths with a `<video>` tag, everything else with `<img>`.

**Pillar 2 (Accessibility, Q4‚ÄìQ10) ‚Äî fully journey-based:** Q4 highlights the language switcher. Q5/Q6 run as ONE continuous on-camera journey (`runLanguageJourney`): the switcher behavior (direct-toggle vs menu-opener, e.g. TAMM's "Choose preferred language" side panel) is discovered LIVE ‚Äî one 1s-hold click, observe, react ‚Äî no off-camera probe. Key hardening (all evaluator-driven, keep intact):
- The control finder only accepts candidates **clickable in the viewport** and prefers the **header region (y < 300)** ‚Äî on Arabic-default sites `a:has-text("ÿπÿ±ÿ®Ÿä")` otherwise matches news articles mid-page.
- `checkTranslated` treats a **URL-only change as NOT a switch** (lang attr, direction, or Arabic-ness of text must also move) ‚Äî an incidental link click must not fake a language switch.
- Menu-opener case: the revealed panel/banner is highlighted whole as "Language Control" (`highlightLanguagePanel`: newly-visible-container diff ‚Üí known panel selectors with relaxed size caps for full-width banners ‚Üí clicked button's ancestor holding ‚â•2 language options).
- Q6 persistence navigates **like a human**: `collectNavTargets` excludes language-toggle links (`/en/`, `/ar/`, hreflang, language-word anchors) and the current page, prefers viewport-visible links sharing the switched locale prefix; if <2 visible it opens the hamburger via `openNavMenu` and clicks drawer links (`target="_self"` forced, `clickWithHighlight`), falling back to direct `goto`. Restore afterwards via `restoreOriginalLanguage` (clears localStorage/sessionStorage AND cookies ‚Äî TAMM persists locale in cookies ‚Äî then reloads + re-dismisses cookie banner).
- Q7 wheel-scrolls down to the example image on video before labeling its alt text. Q8 is a **functional 3-tier check** (1 functional / 0.5 present-not-functional / 0 none): clicks the resize control, then an A+/increase option inside an opened panel, then a font-size **slider** (`TEXT_SLIDER_SELECTORS`, clicks near each end ‚Äî RTL panels put "bigger" on the left; TAMM's is `div[aria-label="slider-sensor"]`). Q9 (screen reader semantics) is detect-only. Q10 clicks the contrast/theme control; if it opens a theme menu (TAMM's moon button `aria-label="header-theme-menu"`), clicks an explicit Dark option (`THEME_OPTION_SELECTORS`); scores 3-tier; narrates absence on video.

**Known site-specific behaviors (regression baseline):** TAMM (tamm.abudhabi): menu-opener language panel, AA-widget font-size slider, theme menu with Dark option, locale persisted in cookies ‚Äî pillar 2 should score 7/7 pass. ADMO (mediaoffice.abudhabi): serves **Arabic by default** (geo), header "EN" direct toggle, navigation only behind a 3-dash burger, legitimately has NO text-resize or colour/theme control ‚Äî Q4‚ÄìQ7, Q9 pass; Q8, Q10 fail.

**TAMM-specific branching:** `isTammHost` (`tamm.abudhabi` / `tamm.ae`) is duplicated in `pillar6-services.ts`, `pillar8-privacy.ts`, `pillar9-livechat.ts`, and `enquiry-grade.ts` (as `isTammDomain`). Pillar 6 classifies the service channel (`entity_website`/`directs_to_tamm`/`navigating_tamm`/`navigating_atlp`) and only evaluates the active branch's Q32/Q33/Q34 variants. Pillar 9 distinguishes own chat / powered-by-TAMM / directs-to-TAMM; auditing TAMM itself is treated as "own chat" (scored normally). Checks must generalize to any entity website ‚Äî don't overfit selectors to TAMM.

**Enquiry email tail (Pillar 10):** on gated submit, a `PendingEnquiry` row is written (`deadlineAt` = +2 UAE business days) and Q56‚ÄìQ66 emit `pending`, marking the job `partial`. Grading logic lives in `enquiry-grade.ts` (pure, no DB) + `enquiry-inbox.ts` (IMAP via `imapflow`/`mailparser` ‚Äî **neither is in `package.json` or `node_modules`**; `npm run build` fails its typecheck on this file, and any runtime use requires installing them first). The background poller referenced in schema comments (`scripts/poll-enquiries.ts`) **does not exist in this tree**; nothing currently resolves `pending` rows.

**Auth:** NextAuth v4 credentials provider (`lib/auth.ts`), bcrypt, JWT sessions; hardcoded dev fallback secret if `NEXTAUTH_SECRET` unset. API routes use `getServerSession(authOptions)` ‚Äî except `stream`, `config`, and `pillars`, which are unauthenticated (and `stream` doesn't scope the audit to its owner). `run-pillar`/`create` scope jobs to the logged-in user. Client-side login redirects are deliberately disabled (`if (false)` in dashboard, empty effect in audit/new) ‚Äî pages render unauthenticated; only the APIs 401.

**Data model (`prisma/schema.prisma`):** `User` ‚Üí `AuditJob` (aggregate score/grade/status: pending|running|partial|complete|failed) ‚Üí `CriterionResult` (one row per question per audit, cascade-deletes) and `PdfReport`; standalone `PendingEnquiry`. **Schema drift:** the single committed migration predates `CriterionResult.isAutomatic/checkType/dependsOn/videoPath`, `AuditJob.acronym`, and the whole `PendingEnquiry` model ‚Äî the live DB was evidently updated via `prisma db push`, so don't rely on `migrate dev` reproducing it cleanly.

**Pillar 1 (Discovery) search flow:** engines are tried in the order Startpage ‚Üí Yahoo ‚Üí Bing ‚Üí Google ‚Üí DuckDuckGo, falling through only when an engine fails validation (captcha/blank/sign-in). Each engine is driven like a person: homepage ‚Üí typed query (`pressSequentially`) ‚Üí Enter, with the direct `?q=` URL as a same-engine fallback; DuckDuckGo's right-rail promos are hidden before evidence. Q1/Q1b tag the matching result anchor (`data-sitecheck-target`), highlight it, click it, and land on the entity site on video. Q1b uses `AuditJob.acronym` when set, else `deriveAcronym` (single-word names and embedded all-caps tokens like "TAMM" are preserved; otherwise word initials). Q3 types the entity name into Google Maps, opens the best-matching result, screenshots with the website field highlighted, and forces `target=_self` before clicking the website link (Maps opens sites in a new tab, which would not appear in the recorded page's video). Pillar 1 keeps its own local `humanScanScroll`/`scrollResultIntoView` (shorter scan gestures) separate from the shared `humanScrollVerify`.

## Known gotchas

- `README.md` is untouched create-next-app boilerplate ‚Äî real docs are this file.
- `puppeteer` is a dependency only for `lib/pdf-generator.ts`; the audit engine is 100% `playwright`.
- Committed generated artifacts exist under `public/reports/` and `public/screenshots/` (stale test data).
- `configWarnings` state in `app/audit/new/page.tsx` is fetched (from `/api/audit/config`, which checks `SERPAPI_KEY`/`GOOGLE_PLACES_API_KEY`) but never rendered; those keys aren't used by the engine.
- `STRICT_LANG_SELECTORS` contain Playwright-only `:has-text()` pseudo-classes ‚Äî they THROW inside `document.querySelector` (callers catch and skip), so browser-side checks effectively use only the attribute/class selectors.
- Playwright `isVisible()` returns true for off-canvas drawer links (non-empty box); require in-viewport coordinates when "clickable by a human" matters.
